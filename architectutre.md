# System Architecture & Flow

## 1. High-Level Flow
1. **Ingestion:** Documents -> Chunking/Parsing -> Indexed into 3 distinct local engines: Dense (Vector), Sparse (BM25), and Open Knowledge Graph (Structured JSON).
2. **Retrieval:** User Query -> Fetches results from all 3 engines -> Reciprocal Rank Fusion (RRF) Scoring -> Top K Combined Chunks.
3. **Generation:** Top K Chunks + Query -> Groq API (Llama 3) -> Draft Answer.
4. **Critique (LangGraph):** Draft Answer + Source Chunks -> Gemini 1.5 Flash API -> Pass/Fail.
5. **Routing:** * If Pass: Return verified answer to user.
   * If Fail: Rewrite query -> Loop back to Retrieval (Max 3 retries).

## 2. Tech Stack
* **Orchestration:** LangGraph, LangChain (Python)
* **Dense Retrieval:** `all-MiniLM-L6-v2` via `sentence-transformers` (Local)
* **Sparse Retrieval:** `rank_bm25` (Local)
* **Graph Retrieval:** Custom Python JSON parser (Markdown-to-Graph)
* **Vector Store:** FAISS or ChromaDB (In-memory/Local SQLite)
* **Generation API:** Groq (`Llama-3-8b-8192`)
* **Critic/Eval API:** Google AI Studio (`Gemini 1.5 Flash`)
* **Frontend/UI:** Streamlit

## 3. Architectural Specifications

### A. The Third Retrieval Engine: Open Knowledge Format (Graph)
To fulfill the three-way benchmark requirement, the system must parse incoming documentation into a hierarchical JSON Graph before indexing.
* **Ingestion Logic:** The parser reads Markdown files, treats Headers (`#`, `##`, `###`) as **Nodes**, and paragraphs/lists under those headers as **Node Content**. Relationships (parent-to-child headers) are mapped as **Edges**.
* **Retrieval Logic:** When a user query matches a specific concept node, the retriever fetches the node content *plus* the content of its immediate parent and child nodes to maintain structural context.

### B. Rigid LangGraph State Definition
The agent must implement the exact state schema using Python's `TypedDict` to govern the cyclical self-healing loop:

```python
from typing import TypedDict, List, Dict, Any

class RagAgentState(TypedDict):
    original_query: str          # The initial user prompt
    current_search_query: str    # The optimized query (rewritten if healing occurs)
    retrieved_chunks: List[Dict[str, Any]] # Combined chunks from all 3 engines
    draft_answer: str            # Current response generated by Groq
    critique_score: float        # Groundedness evaluation score from Gemini
    critique_feedback: str       # Reason for failure if model hallucinated
    loop_count: int              # Counter to break the loop at Max 3 retries
```

### C. Golden Dataset & Evaluation Schema
The `golden_dataset.json` used by the CI/CD pipeline and the Ragas evaluation library must adhere strictly to this format:

```json
[
  {
    "question": "What is the exact memory limitation of the local host machine?",
    "ground_truth": "The local host machine has a hard cap of 16GB shared RAM.",
    "expected_context": [
      "The host machine has a hard cap of 16GB shared RAM. DO NOT attempt to load large LLMs locally."
    ]
  }
]
```

The Ragas/TruLens evaluation node must explicitly measure and log these four native metrics:
1. `faithfulness`: Measures if the answer is derived *only* from the context (detects hallucinations).
2. `answer_relevance`: Measures if the answer directly addresses the user's question.
3. `context_precision`: Measures if the RRF algorithm successfully ranked the most important chunks at the top.
4. `context_recall`: Measures if the retrieval engines fetched all the necessary information required to answer the ground truth.

## 4. Core Mathematical Formulas

### A. Reciprocal Rank Fusion (RRF)
Used in the Hybrid Retrieval Engine to combine bounded scores (Dense Cosine Similarity) and unbounded scores (Sparse BM25). It ranks documents based on their positional standing rather than raw scores. **Note:** $R$ officially equals 3 for this project ($R = \{\text{BM25}, \text{Dense Vector}, \text{Knowledge Graph}\}$).

$$RRF(d) = \sum_{r \in R} \frac{1}{k + rank_r(d)}$$

**Variables:**
* $d$: The specific document chunk being scored.
* $R$: The set of retrieval methodologies (3 engines).
* $rank_r(d)$: The ranked position (1st, 2nd, 3rd) of document $d$ within retriever $r$'s results.
* $k$: A smoothing constant. **Default to 60** for this project.

### B. Perplexity (Evaluation Metric)
Used by the LLM-as-a-judge to measure how well the model predicts the generated text based on the retrieved context. 

$$PP(W) = P(w_1, w_2, ..., w_N)^{-\frac{1}{N}}$$

**Variables:**
* $W$: The sequence of words in the generated answer.
* $N$: The total number of words in the sequence.
* $P(w_i)$: The probability of the $i$-th word given the preceding words.

### C. Cosine Similarity (Dense Retrieval Fallback)
Used by the Vector Database (FAISS/Chroma) to measure the angular distance between the user's query vector and the document's chunk vector.

$$\text{Cosine Similarity}(A, B) = \frac{A \cdot B}{\Vert{}A\Vert{} \Vert{}B\Vert{}}$$

**Variables:**
* $A$: The embedded vector of the user's search query.
* $B$: The embedded vector of the document chunk.
